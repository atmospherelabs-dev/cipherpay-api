Part 2

Backend Interaction: RPC & APIs

Three products, three ways to talk to Zcash

CipherScan ↔ Zebra

RocksDB Direct Read

  • Read Zebra's database directly
  • No RPC overhead
  • Full block + transaction data

gRPC Streaming

  • Real-time block events
  • Replaces 30-second polling
  • Sub-second latency

Key insight: Polling = 0-30s latency. gRPC stream = instant. For payment detection, this matters.

CipherPay ↔ CipherScan

Service-to-service via WebSocket

// ws.rs — Connect to CipherScan WebSocket
pub async fn run(ws_url: String, service_key: String,
                 tx: mpsc::Sender<MempoolPush>) {
    loop {
        let mut stream = connect(&ws_url, &service_key).await?;

        // Subscribe to raw mempool transactions
        let sub = json!({"subscribe": "raw_mempool"});
        stream.send(Message::Text(sub.to_string())).await?;

        // Each message = a new mempool transaction with raw hex
        while let Some(msg) = stream.next().await {
            if let Some(push) = parse_mempool_tx(&text) {
                tx.send(push).await?;  // Forward to scanner
            }
        }
    }
}

Auth: X-Service-Key header · Auto-reconnect with backoff

Zipher ↔ lightwalletd

Compact Blocks via gRPC

  • Downloads compact block data
  • Trial decryption happens client-side
  • Server never sees your payments
  • Standard Zcash wallet protocol

Different Trust Model

  • CipherPay: server decrypts (IVK trusted to service)
  • Zipher: client decrypts (keys never leave device)
  • Same crates, same math, different architecture

Takeaway: You don't need a full node. Pick the right data source:
RocksDB direct · gRPC stream · REST API · lightwalletd

Payment Flow Design

From invoice creation to webhook delivery

Step 1-2: Merchant → Invoice

1. Merchant registers with their Incoming Viewing Key (UIVK)
CipherPay stores only the IVK — never the full viewing key, never the spending key
2. Invoice created — diversified payment address derived from IVK
Each invoice = unique address. Zcash URI + QR code generated.

Step 3-4: Payment → Detection

3. Buyer pays — shielded transaction hits the mempool
4. Mempool detection — CipherScan pushes raw tx hex via WebSocket
CipherPay trial-decrypts every output against all active IVKs
// Trial-decrypt Orchard outputs with pre-computed keys
pub fn try_decrypt_with_keys(raw_hex: &str, keys: &CachedKeys)
    -> Result<Vec<DecryptedOutput>>
{
    let tx = Transaction::read(&mut cursor, BranchId::Nu6_2)?;
    let bundle = tx.orchard_bundle()?;

    for action in bundle.actions() {
        // Try both External and Internal scopes
        if let Some((note, _, _)) = try_note_decryption(
            &keys.domain, &keys.ivk, action
        ) {
            outputs.push(DecryptedOutput {
                amount_zatoshis: note.value().inner(),
                memo: extract_memo(action, &keys.fvk),
                // ...
            });
        }
    }
    Ok(outputs)
}

Result: sub-second detection. Match found = payment detected.

Step 5-7: Confirmation → Webhook

5. Block confirmation — scanner confirms the transaction on-chain
Captures fiat rate at confirmation time (accounting compliance)
6. Webhook fires — HMAC-signed, with delivery-ID deduplication
7. Merchant's system marks the order as paid
// HMAC-SHA256 webhook signature
fn sign_payload(secret: &str, timestamp: &str, payload: &str) -> String {
    let message = format!("{}.{}", timestamp, payload);
    let mut mac = HmacSha256::new_from_slice(secret.as_bytes())?;
    mac.update(message.as_bytes());
    hex::encode(mac.finalize().into_bytes())
}

// Retry schedule: 1min, 5min, 25min, 2h, 10h
fn retry_delay_secs(attempt: i64) -> i64 {
    match attempt {
        1 => 60, 2 => 300, 3 => 1500, 4 => 7200, _ => 36000,
    }
}

Design Decisions


Why IVK-only?

Principle of least privilege.
We detect payments, nothing more.


Why diversified addresses?

Future-proof for Tachyon.
No memo-matching fragility.

Why HMAC webhooks?

Reliable, real-time, verifiable.
Delivery-ID prevents double-processing.


Why capture fiat rates?

Tax compliance needs fiat value
at the moment of payment.
